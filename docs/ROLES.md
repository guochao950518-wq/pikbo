# Pikbo AI 分工（老板定 · 唯一权威）

**产品：** 潮玩垂直 AI 创作平台（pikbo.ai）  
**仓库：** https://github.com/guochao950518-wq/pikbo  
**原则：** 严格按车道推进；**有交叉先沟通再动手**（写进 `docs/DISPATCH.md` 或 `docs/HANDOFF.md` 一条「交叉请求」）。

---

## 总表

| Agent | 角色 | 主交付物 | 不负责 |
|-------|------|----------|--------|
| **Claude** | **工程实现主力** | 全部代码编写、前后端、架构、重构、长上下文维护 | 不独自定商业规则/PRD；不空谈增长策略 |
| **GPT (Codex)** | **产品与结构** | PRD、数据结构、业务逻辑、API 设计、流程、商业规则、分析 | 不大批量写业务代码（可小样/伪代码辅助说明） |
| **Grok** | **创意 · 差异化 · 增长** | 潮玩功能脑暴、定位、增长/社区/病毒、市场洞察、prompt 优化、趋势 | 不抢主写业务代码；不替代 PRD |

---

## Claude — 代码与工程

**做：**
- 前后端开发（Next.js、API、组件、样式实现）
- 系统架构、模块边界、重构、类型与测试
- 长上下文下的代码库维护、修 bug、合冲突中的实现侧
- 按 GPT 的 PRD/API 规格 **实现**；按 Grok 的 prompt/玩法规格 **落地**

**不做：**
- 不自己改定价商业规则而不读 GPT 规格
- 不为了「好看」推翻已冻结的产品流程（先找 GPT）

**分支：** `agent/claude/<topic>` · commit `[claude]`  
**建议目录：** 主仓或 `pikbo-claude-ui` 仅当隔离 UI 大改；**代码权威以合入 `main` 为准**

---

## GPT — 产品规划与结构化

**做：**
- 功能规格（PRD）、用户故事、验收标准
- 数据结构、状态机、积分/订阅商业规则文档
- API 契约草案（路径、字段、错误码）—— **实现由 Claude**
- 系统流程图、漏斗与数据分析框架
- 诚实口径与单位经济的产品侧说明

**交付位置（约定）：**
- `docs/prd/` · `docs/api/` · `docs/business/`（可新建）
- 更新 `docs/STATUS.md` 任务与验收
- 不直接改 `app/api/**` 生产逻辑，除非 Claude 明确请你补丁且极小

**分支：** `agent/gpt/<topic>` · commit `[gpt]`

---

## Grok — 创意 · 差异化 · 增长

**做：**
- 潮玩垂直功能脑暴、与 HF 等竞品的差异化定位
- 增长策略、社区玩法、病毒传播、上线节奏建议
- 市场洞察、prompt 工程（`lib` 里 prompt 模板的**文案/策略**建议；大改由 Claude 合）
- 实时趋势结合、DISPATCH 协调、合流冲突时的**产品优先级判断**
- 老板沟通、进度简报

**不做：**
- 不成为默认「全栈写手」（历史引擎活已合 main；新功能实现归 Claude）
- 不独自定 API 字段而不经 GPT 规格

**分支：** `agent/grok/<topic>` · commit `[grok]`（协调文档、策略、prompt 草案为主）

---

## 交叉需求协议

1. **谁先写「交叉请求」** 到 `docs/DISPATCH.md` → Cross requests  
2. 另一方 **确认后再动**  
3. 例：
   - Grok 提出「盲盒抽奖社区」→ GPT 写 PRD → Claude 实现  
   - GPT 定「积分按秒计费」→ Claude 改 `lib` + API  
   - Claude 发现架构需改契约 → 先找 GPT 更新规格  

---

## 与旧分工的关系

| 旧 | 新 |
|----|-----|
| Grok 写引擎/合流代码 | Claude 写代码；Grok 协调+增长 |
| GPT 只写转化文案 | GPT 升为 PRD/结构/商业规则 |
| Claude 只做 UI 审美 | Claude **全栈实现**（含 UI 与后端） |

旧代码保留；**新任务严格按新表。**
